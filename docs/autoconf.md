# autoconf/automake を使用して bdwgc をビルドする

`autoconf`、`automake`、`libtool`に基づく GNU スタイルのビルドをサポートします。
これは、ほぼ完全に _Tom Tromey_ の`gcj`の作業に基づいています。

次のように入力して`configure`を再生成します (`configure.ac`より古い場合)。

```sh
./autogen.sh
```

次のように入力して、ライブラリをビルド、テスト、インストールします。

```sh
./configure && make && make check && make install
```

このプロセスの利点は次のとおりです。

1. 最終的には、
使用する適切なコンパイラなど。おそらく、場合によってはすでにそうなっているでしょう。

2. デフォルトの bdwgc パラメータの適切なセットを自動的に計算しようとします。
プラットフォーム用 (マルチスレッドのサポートなど)。より簡単な方法を提供します
他のパラメータのいくつかを設定する方法。

3. GNU スタイルのビルド プロセスを使用して、他のプロジェクトとより適切に統合します。

4. 動的ライブラリと静的ライブラリの両方を構築します。

既知の欠点は次のとおりです。

1. ビルド スクリプトははるかに複雑で、デバッグが困難です。
理解します（ただし、それらはほとんど標準的です）。いくつかあるかもしれない
余計なもの。

2. おそらく、まだすべての UNIX 系プラットフォームで動作するとは限りません。それはおそらく
他のものでは決して機能しません。

3. スクリプトがまだ完成していない可能性があります。標準的な GNU ターゲットの一部
まだうまく機能していない可能性があります。  (修正/追加は大歓迎です。)

ディストリビューションには、`configure`を実行するために必要なすべてのファイルが含まれている必要があります。
`make`、および派生ファイルを再生成するために必要なソース。

ディストリビューションには、によって生成される`Makefile`が含まれていないことに注意してください。
__インラインコード0__。  ディストリビューションには、そうではない`Makefile.direct`も含まれています。
常に生成されたものと等価です。

構成する重要なオプションのいくつかは次のとおりです。

*`--prefix=PREFIX`: アーキテクチャに依存しないファイルを`PREFIX`にインストールします
(UNIX 系プラットフォームではデフォルトで`/usr/local`パス)。

*`--exec-prefix=EPREFIX`: アーキテクチャに依存するファイルを`EPREFIX`にインストールします
(デフォルトでは`prefix`と同じ)。

*`--enable-threads=TYPE`: マルチスレッド パッケージを選択します。

*`--enable-gc-debug`: ポインタのバックトレースなどの完全なサポートが含まれます。

UNIX 系プラットフォームでは、`--prefix`(または`--exec-prefix`、または 1 つが設定されている場合を除く)
よりあいまいなオプション)、`make install`は`libgc.a`をインストールします。
`libgc.so`をそれぞれ`/usr/local/lib`と`/usr/local/bin`にすると、
通常、`make install`を root として実行する必要があります。

マルチプロセッサの並列マーキングをオフにすることは推奨されません。
ターゲット プラットフォームの機能にはいくつかの問題が存在します。
