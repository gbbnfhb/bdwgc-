# C/C++ インターフェース

多くのプラットフォームで、シングルスレッドのガベージ コレクター ライブラリを構築できます。
プラグイン`malloc`の代替として機能します。 (これを使って構築します
`-D REDIRECT_MALLOC -D IGNORE_FREE`.) 多くの場合、これが最善の対処方法です。
オブジェクトをリークしたり、早期に解放したりするサードパーティのライブラリ。
`-D REDIRECT_MALLOC`は主に古いコードを簡単に適応させることを目的としています。
新規開発のためではありません。

新しいコードでは、以下で説明するインターフェイスを使用する必要があります。

コードは GC ライブラリに対してリンクする必要があります。ほとんどの UNIX プラットフォームでは、
コレクターの構築方法に応じて、これは`libgc.a`または`libgc.so`になります。

以下に、ガベージ コレクターへの標準 C インターフェイスについて説明します。
これはインターフェイスの完全な定義ではありません。最も重要なことだけを説明します

使用の。これは、`gc.man`ファイル内の情報と多少重複します。完全な
インターフェースは`gc.h`ファイルに記述されています。

クライアントには`gc.h`ファイル (つまり、`gc_config_macros.h`ではなく、
`gc_pthread_redirects.h`、`gc_version.h`ファイル)。マルチスレッドの場合
コード、`gc.h`ファイルは、スレッド ヘッダー ファイルの後、およびその後にインクルードする必要があります。
`GC_THREADS`マクロを定義しています。`gc.h`ファイルは、使用するファイルに含める必要があります。
スレッド プリミティブは再定義されるため、GC またはスレッド プリミティブのいずれか
多くのプラットフォームで GC と連携します。

スレッド ユーザーは、多くのプラットフォームでオブジェクトが到達可能であることにも注意する必要があります。
スレッドローカル変数からのみ、早期に再利用できます。したがって、オブジェクト

`void *GC_MALLOC(size_t _bytes_)`- _bytes_ の割り当てとクリア

コレクタ。 (ただし、`GC_MALLOC_UNCOLLECTABLE`を参照してください。コレクターの構築
`-D REDIRECT_MALLOC_UNCOLLECTABLE`を使用すると、多くの場合これを回避できます。)
`GC_MALLOC`は、デフォルトで`GC_malloc`を呼び出すマクロ、または`GC_DEBUG`の場合は
`gc.h`ファイルがインクルードされる前に定義されており、チェックするデバッグ バリアント

`void *GC_MALLOC_ATOMIC(size_t _bytes_)`- _bytes_ を割り当てます
ストレージの。 _bytes_ に比例する (償却された) 時間が必要です。結果として得られる
オブジェクトは参照されなくなると自動的に割り当てが解除されます。クライアント
結果のオブジェクトにはポインターが含まれないことを約束します。記憶
クリアされていない。これは、文字列、浮動小数点を割り当てるための推奨される方法です。
配列、ビットマップなど。ポインタの位置に関するより正確な情報は、
`gc_typed.h`ファイルのインターフェイスを使用してコレクターに通信されます。

`void *GC_MALLOC_UNCOLLECTABLE(size_t _bytes_)`- 同一
`GC_MALLOC`に変換されます。ただし、結果のオブジェクトは自動的には生成されません。

ガベージコレクタ可能なメモリへのポインタのオブジェクト (ブロック自体であっても)

`void *GC_REALLOC(void * _old_object_, size_t _new_bytes_)`- 新しいものを割り当てます

新しいオブジェクト。古いオブジェクトは、必要に応じて所定の位置で再利用されます。オリジナルなら
オブジェクトが`GC_MALLOC_ATOMIC`で割り当てられた場合、新しいオブジェクトは
同じ制約。収集不可能なオブジェクトとして割り当てられた場合、新しい
オブジェクトは収集できず、古いオブジェクト (異なる場合) の割り当てが解除されます。

`void *GC_REALLOCF(void * _old_object_, size_t _new_bytes_)`- と同じ
`GC_REALLOC()`ですが、次の場合には`_old_object_`も割り当て解除されます。
割り当ての失敗。

`void GC_FREE(void * _object_)`- _object_ の割り当てを明示的に解除します。
通常、小さな収集品には役に立ちません。

`void *GC_MALLOC_IGNORE_OFF_PAGE(size_t _bytes_)`および
`void *GC_MALLOC_ATOMIC_IGNORE_OFF_PAGE(size_t _bytes_)`- 類似
それぞれ`GC_MALLOC`と`GC_MALLOC_ATOMIC`に変換されます。ただし、クライアント

## C++ インターフェース

C++ インターフェイスは、C インターフェイス上の薄い層として実装されます。
残念ながら、この薄い層は変化に非常に敏感であるようです
C++ 実装では、特にグローバルを置き換えようとするため、
`::new`演算子。十分に標準化されていないように見えます。あなたの
プラットフォームでは、このレイヤーで若干の調整が必要な場合があります (`gc_badalc.cc`、
`gc_cpp.cc`、`gc_cpp.h`、および場合によっては`gc_allocator.h`ファイル)。このような変化
コレクターの内部構造を理解する必要はありませんが、必要な場合があります。
プラットフォームをよく理解すること。 (移植性を向上させるパッチは、
いらっしゃいませ。ただし、あるプラットフォームを修復することで別のプラットフォームを破壊するのは簡単です。)

C++ からのコレクターの使用も、次のような理由から複雑になります。
C++ でメモリを割り当てるための多くの_標準_ 方法。デフォルトの`::new`演算子、
デフォルトの`malloc`とデフォルトの STL アロケータは、そうでないメモリを割り当てます。
ガベージ コレクトされ、通常はコレクタによって_追跡_されません。これはつまり
これらのデフォルトのアロケータによって割り当てられたメモリ内のポインタは、
コレクターに見られました。ポインタによってのみ参照されるガベージコレクタブルメモリ
このようなデフォルトで割り当てられたオブジェクトに格納されているオブジェクトは、時期尚早に再利用される可能性があります。
コレクターによる。

ガベージコレクタ可能なメモリを確保するのはプログラマの責任です。
次のいずれかに格納されたポインタによって参照されます。

* プログラム変数
* ガベージコレクションされたオブジェクト
* 未収集だが_追跡可能な_オブジェクト

追跡可能なオブジェクトは必ずしもコレクターによって回収されるわけではありませんが、
収集可能なオブジェクトへのポインタをスキャンします。通常は割り当てられます
上で説明したように、`GC_MALLOC_UNCOLLECTABLE`によって、およびいくつかのインターフェイスを通じて
以下に説明します。

ほとんどのプラットフォームでは、コレクターが実行中に正しくトレースできない可能性があります。
例外オブジェクト。したがって、例外としてスローされるオブジェクトはポイントのみを指定する必要があります。
さもなければ到達可能なメモリに。これも適切な修復が必要な別のバグです
プラットフォームフックが必要です。

収集可能なオブジェクトが適切に参照されていることを確認する最も簡単な方法
収集可能なオブジェクトのみを割り当てることです。これには、すべての割り当てが必要です。
次のインターフェイスのいずれかを通過し、それぞれが
標準の C++ 割り当てメカニズム。これには、すべての STL が必要であることに注意してください。
コンテナは`gc_allocator`を使用して明示的にインスタンス化されます。
