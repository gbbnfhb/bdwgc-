## 木

この説明の残りの部分では、使用される 2 レベルのデータ構造に焦点を当てます。
上位ビットと中位ビットをブロック記述子にマッピングします。

上位ビットは`GC_top_index`へのインデックスとして使用されます (実際には
`GC_arrays._top_index`) 配列。各エントリは`bottom_index`データを指します
構造。この構造の大部分は、インデックス付きの配列`index`で構成されます。
候補ポインタの中間ビットによって決定されます。`index`配列には、
実際の`hdr`ポインター。

したがって、ポインタ ルックアップは主に少数のメモリ参照で構成されます。
そしてかなり高速になる可能性があります:

1. 適切な`bottom_index`ポインターが`GC_top_index`で検索されます。
候補ポインタの上位ビットに基づきます。

2. 適切な`hdr`ポインターが`bottom_index`内で検索されます。
中間ビットに基づく構造。

3. ブロック レイアウト マップ ポインターが`hdr`構造体から取得されます。 （これ
ブロック レイアウト マップを共有しようとしているため、メモリ参照が必要です。)

4. オブジェクトの先頭までの変位は、
上の地図。

スペースを節約するために、実際にはすべての`GC_top_index`エントリがポイントするわけではありません。
個別の`bottom_index`構造に変換します。対応するアドレスがない場合
上位ビットがヒープの一部である場合、エントリは`GC_all_nils`を指します。
`NULL``hdr`ポインターのみで構成される単一の`bottom_index`構造体。

`bottom_index`構造には、`hdr`よりもわずかに多くの情報が含まれます
ポインタ。`asc_link`フィールドは、すべての`bottom_index`構造をリンクするために使用されます。
高速移動の場合は昇順で。このリストが指摘しているのは、
__インラインコード0__。これは`key`フィールドを使用して維持されます。
`bottom_index`に対応する上位ビットが含まれます。

## 64 ビット アドレス

64 ビット アドレスの場合、この図は少し複雑になります。
実際、インデックス構造の 1 つが巨大なものになる必要があります。
アドレス空間全体を 2 レベルのツリーで構成します。私たちはこれに対処します
`GC_top_index`を単純な配列ではなく、連鎖ハッシュ テーブルに変換します。これは追加します
`hash_link`フィールドを`bottom_index`構造体に追加します。

ハッシュ関数は上位ビットを削除することで構成されます。これは安いです
を計算し、ヒープが存在する場合に衝突が発生しないことを保証します。
連続していて、過度に大きくない。
